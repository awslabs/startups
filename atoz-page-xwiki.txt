= GCP to AWS Migration — AWS Startups Migrate =

Migrate from Google Cloud Platform (GCP) to AWS with AI-guided planning and execution. AWS Startups Migrate generates a complete GCP-to-AWS migration plan — service mappings, production-ready Terraform, three-tier cost estimates, and a phased runbook — in a single session. It also migrates AI workloads from OpenAI or Google Gemini to Amazon Bedrock with honest model-by-model pricing comparisons.

== Why Startups Struggle to Migrate from GCP to AWS ==

For startups running on Google Cloud, migrating to AWS sounds worthwhile — but four things keep them stuck:

* **Assessment Effort** — Discovery alone takes 3–4 weeks of back-and-forth before you even have a GCP-to-AWS migration plan, and many deals die in that window.
* **Data Sharing Anxiety** — Startups won't hand over their GCP bills, repos, or infrastructure code. Without that data, cost comparisons are guesswork and AWS service recommendations miss the mark.
* **Cost Uncertainty** — No easy way to compare AWS vs. current GCP spend without doing the work first.
* **Execution Gap** — No clear path from migration plan to actually running on AWS.

== How GCP to AWS Migration Works — The 6-Phase Workflow ==

AWS Startups Migrate runs a structured 6-phase assessment that produces a complete migration plan:

=== Phase 1: Discover ===

The tool scans your Terraform files, application source code, and/or billing data. It detects GCP resources, AI models, agentic frameworks, tools, and orchestration patterns — building a full resource inventory. You can also provide just a GCP billing export (CSV/JSON) without any code or Terraform — the tool produces a lower-confidence but usable plan from billing data alone.

=== Phase 2: Clarify ===

Targeted questions about traffic patterns, availability needs, compliance requirements, target region, cutover strategy, and growth projections — only asking what it can't infer from your data.

=== Phase 3: Design ===

Maps GCP services to AWS equivalents with confidence scoring and rationale. For AI workloads: selects optimal Bedrock models with honest pricing comparison. For agentic workloads: recommends one of three migration paths (retarget, AgentCore Harness, or Strands Agents).

=== Phase 4: Estimate ===

Calculates monthly AWS costs across three tiers — Premium, Balanced, and Optimized — using real-time AWS Pricing API data. Produces a side-by-side comparison against your current GCP/OpenAI spend.

=== Phase 5: Generate ===

Creates runnable migration artifacts: validated Terraform (vpc.tf, compute.tf, database.tf, security.tf, baseline.tf), numbered migration scripts, AI provider adapters, a phased migration guide, and an executive summary report. The security baseline (baseline.tf) is always generated and includes GuardDuty, CloudTrail, IMDSv2 enforcement, ECR scanning, EBS encryption, and budget alerts.

=== Phase 6: Feedback (optional) ===

Collects anonymized feedback to improve the tool.

== GCP to AWS Service Mapping ==

AWS Startups Migrate dynamically maps GCP services to their optimal AWS equivalents based on your actual workload characteristics:

|=GCP Service|=AWS Equivalent|=When Recommended
|Cloud Run|AWS Fargate|General containerized workloads
|GKE|Amazon EKS|Kubernetes experience, multi-cloud portability
|GKE|Amazon ECS / Fargate|Simplicity preference, managed containers
|Cloud SQL (PostgreSQL/MySQL)|Amazon Aurora|Variable traffic, high availability requirements
|Cloud SQL (PostgreSQL/MySQL)|Amazon RDS|Steady traffic patterns, single-AZ
|Cloud Functions|AWS Lambda|Serverless functions
|Cloud Storage|Amazon S3|Object storage
|Pub/Sub|Amazon SNS / SQS / EventBridge|Messaging and event routing
|Memorystore|Amazon ElastiCache|In-memory caching
|Cloud Load Balancing|ALB / NLB|Load balancing
|Cloud DNS|Amazon Route 53|DNS management
|Cloud Armor|AWS WAF + Shield|Web application firewall
|Secret Manager|AWS Secrets Manager|Secrets management
|VPC|Amazon VPC|Networking
|Filestore|Amazon EFS|File storage
|BigQuery|//Deferred — specialist engagement//|Analytics workloads require expert consultation

**Note on BigQuery:** The tool deliberately defers BigQuery migrations to specialist engagement rather than auto-mapping to Redshift or Athena. This ensures analytics workloads get the expert attention they require.

== AI Migration: OpenAI & Gemini to Amazon Bedrock ==

Migrate AI workloads to AWS alongside or independent of infrastructure:

* **Dynamic model mapping** — Your specific models mapped to optimal Bedrock equivalents (Claude, Nova, Titan, Llama) with honest per-model stay-or-migrate recommendations based on current pricing. Supports migrations from OpenAI, Google Gemini, and direct Anthropic API usage.
* **Three-tier cost analysis** — Real-time pricing from AWS Pricing API with side-by-side comparison against your existing OpenAI/Gemini spend
* **Code migration** — Your actual API calls converted to Bedrock Converse API format with your model IDs, tool names, system prompts, and target region
* **Incremental migration support** — A/B test with Bedrock per-invocation using shadow mode, cut over when confident
* **Regional validation** — Live availability checks before committing to a design (catches "this model isn't in your target region" before you deploy)

=== Agentic Framework Migration ===

For startups building with agentic architectures, the tool detects your full stack — agents, tools, orchestration patterns, memory layers, and multi-model pipelines — then recommends one of three paths:

|=Migration Path|=Best For|=Effort
|**Retarget**|Keep your framework (LangChain, CrewAI, etc.), swap models to Bedrock|Low
|**AgentCore Harness**|Config-based managed agents — declare an agent in 3 API calls|Medium
|**Strands Agents**|AWS-native multi-agent SDK (open-source, powers AgentCore internally)|Medium–High

**Supported frameworks:** LangGraph, CrewAI, AutoGen, OpenAI Agents SDK, LangChain, LlamaIndex, LiteLLM, OpenRouter, MCP servers, and custom agent loops.

**Agent architectures detected:** Single agent, hierarchical, swarm, graph, and sequential orchestration.

== Ideal Startup Use Cases for GCP to AWS Migration ==

**Best Fit:**

* Startups running infrastructure on GCP (Cloud Run, Cloud SQL, GKE, Cloud Functions, Cloud Storage, VPC)
* Any startup using OpenAI or Gemini APIs — regardless of where their infrastructure lives
* Teams building with agentic frameworks: LangChain, LangGraph, CrewAI, AutoGen, OpenAI Agents SDK
* Workloads: containerized web apps, APIs, AI/LLM inference, agentic workflows, data pipelines

**Not a good fit (today):**

* Legacy monoliths or heavy on-prem dependencies
* Infrastructure on Azure or Oracle Cloud (GCP and Heroku [coming soon] only)
* BigQuery-heavy analytics workloads (deferred to specialist engagement)
* Startups with no GCP infrastructure and no OpenAI/Gemini API usage
