# Generate Phase: Migration Scripts

> Self-contained fragment. Always runs. Produces numbered migration scripts
> in `$MIGRATION_DIR/scripts/`. ALL scripts default to dry-run mode and
> require `--execute` for destructive actions.

**Execute ALL steps in order. Do not skip or optimize.**

---

## Common Script Header

Every script begins with:

```bash
#!/usr/bin/env bash
# <Script Title> — Vercel-to-AWS Migration
#
# Purpose: <one-line description>
# Prerequisites: <list>
# Dry-run: This script runs in dry-run mode by default. Pass --execute to apply.
# Rollback: <how to undo>
#
# Generated by the vercel-to-aws migration skill.

set -euo pipefail

EXECUTE=false
if [[ "${1:-}" == "--execute" ]]; then
  EXECUTE=true
  shift
fi
```

## Step 1: Emit `scripts/01-validate-prerequisites.sh`

Checks before any migration step runs:

- AWS CLI installed and configured (`aws sts get-caller-identity`)
- Terraform >= 1.5 installed
- Target region accessible
- Required IAM permissions (bedrock if AI, ecs/ecr if Fargate, lambda if
  serverless, rds if database)
- Docker installed (if Outcome B — container builds needed)
- `jq` installed (used by other scripts)

Output: pass/fail summary with actionable fix instructions for each failure.

---

## Step 2: Emit `scripts/02-migrate-secrets.sh`

Migrates Vercel environment variables to AWS Secrets Manager:

- Reads `discovery.json.env_var_names[]` (names only — values were never
  persisted per Requirement 1.6)
- For each name, prompts the user for the value (or reads from a provided
  env file)
- Creates the secret in Secrets Manager (or SSM Parameter Store based on
  naming convention — secrets with `_KEY`, `_SECRET`, `_TOKEN` suffixes go
  to Secrets Manager; others to SSM Parameter Store)
- Dry-run: prints what would be created without creating anything
- Rollback: `aws secretsmanager delete-secret --secret-id <name> --force-delete-without-recovery`

---

## Step 3: Emit `scripts/03-migrate-database.sh` (If Postgres Detected)

Conditional: only when `discovery.json.peripherals[]` contains `"postgres"` (the schema's lowercase type value; match case-insensitively).

Select migration tool based on `clarify-answers.json.Q7_database_size`:

- **< 10 GB (pg_dump):** `pg_dump` from source -> `psql` restore to RDS
- **>= 10 GB (DMS):** Create DMS replication instance, source/target
  endpoints, and replication task for zero-downtime migration

Both paths include:

- Connection string templating (source from Vercel, target from Terraform
  output)
- Pre-migration validation (schema compatibility check)
- Post-migration row count comparison
- Dry-run: prints the plan and validates connections without transferring data
- Rollback: "drop and re-create the RDS instance from the final snapshot"

---

## Step 4: Emit `scripts/04-build-and-push.sh` (If Outcome B)

Conditional: only when `recommendation.json.outcome == "B"`.

Builds and pushes the container image to ECR:

```bash
# Build for ARM64 (Graviton) — matches the Fargate task definition
docker buildx build --platform linux/arm64 -t $ECR_REPO:$TAG .

# Authenticate to ECR
aws ecr get-login-password --region $REGION | docker login --username AWS --password-stdin $ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com

# Push
docker push $ECR_REPO:$TAG
```

- Dry-run: builds the image locally without pushing
- Rollback: delete the ECR image tag

---

## Step 5: Emit `scripts/05-dns-cutover.sh` (If Outcome A or B)

Conditional: only for full-app migrations (Outcome A or B).

DNS migration procedure:

1. Lower TTL on current DNS records (to 60s) — at least 24h before cutover
2. Wait for TTL propagation
3. Update DNS to point at CloudFront/ALB (`CNAME` or `ALIAS` record)
4. Verify resolution (`dig` + `curl` health check)
5. Monitor error rates for 30 minutes
6. Restore normal TTL

- Dry-run: prints current DNS records and the planned changes, validates
  the target endpoint is reachable, but does NOT modify DNS
- Rollback: explicit commands to revert DNS to Vercel's CNAME target

```bash
# Rollback — revert DNS to Vercel
# aws route53 change-resource-record-sets ... (if using Route 53)
# Or update your DNS provider to point back to: cname.vercel-dns.com
```

---

## Step 6: Emit `scripts/06-validate-migration.sh`

Post-migration health checks (always emitted):

- HTTP(S) probe against new endpoints (CloudFront domain or ALB DNS)
- Status code validation (expect 200 on key routes)
- Response header checks (`x-amz-cf-id` present for CloudFront)
- SSL certificate validation
- Response time comparison (flag if > 2x slower than Vercel baseline)
- Database connectivity test (if Postgres migrated)

Output: pass/fail summary with specific failure details.

## Output Contribution for Parent Orchestrator

All scripts in `$MIGRATION_DIR/scripts/`. The always-emitted set is
`01-validate-prerequisites.sh`, `02-migrate-secrets.sh`,
`06-validate-migration.sh`. Conditional scripts are emitted based on
recommendation outcome and detected peripherals.

## Error Handling

| Error Category                          | Behavior                                                                                                    |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `discovery.json.env_var_names` is empty | Still emit `02-migrate-secrets.sh` but with a note that no env vars were detected; the script exits cleanly |
| Database size unknown from Clarify      | Default to pg_dump (conservative, works for all sizes, just slower for large databases)                     |
