= How to Migrate from GCP to AWS Using the AWS Startup Advisor Plugin =

Migrate your startup from GCP to AWS directly from your IDE. The **AWS Startup Advisor plugin** works with Kiro, Claude Code, Cursor, and Codex to analyze your codebase, infrastructure, and billing data ‚Äî then generates a complete GCP-to-AWS migration plan with Terraform templates, service mappings, architecture diagrams, and cost estimates.

**Your data stays local** ‚Äî the plugin runs analysis within your coding agent. No code or billing data is uploaded to AWS servers.

== Quick Start: Migrate GCP to AWS in 5 Minutes ==

=== 1. Install the Plugin ===

**Kiro / Cursor (VS Code-based):**

{{code}}
1. Open Extensions panel (Ctrl+Shift+X)
2. Search "AWS Startup Advisor"
3. Click Install
{{/code}}

**Claude Code:**

{{code}}
Visit https://claude.com/plugins/aws-startup-advisor and click "Add to Claude Code"
{{/code}}

**Codex (via agent-plugins-for-aws):**

{{code}}
/plugin install migration-to-aws@agent-plugins-for-aws
{{/code}}

=== 2. Start a Migration ===

Open your terminal or chat panel and type:

{{code}}
Start a migration from GCP to AWS
{{/code}}

or more specifically:

{{code}}
Analyze my GCP infrastructure and create a migration plan to AWS
{{/code}}

**Other prompts that work:**

* "Migrate my Terraform from GCP to AWS"
* "Map my GCP services to AWS equivalents"
* "How much would my GCP workload cost on AWS?"
* "Migrate my OpenAI usage to Amazon Bedrock"
* "Create a Terraform plan for migrating from GKE to EKS"

=== 3. Provide Your Data ===

The plugin will ask for what it needs. You can provide any combination of:

|=Data Source|=How to Provide|=What It Enables
|**Terraform / Pulumi files**|Auto-detected in workspace|Service detection, IaC generation
|**Kubernetes manifests**|Auto-detected in workspace|Container workload mapping
|**GCP billing export (CSV/JSON)**|Upload or point to file path|Cost comparison, right-sizing
|**OpenAI usage CSV**|Upload or point to file path|AI cost comparison, model mapping
|**GitHub repo**|Already connected via IDE|Code pattern analysis, SDK detection

**You don't need all of these.** The plugin works with whatever you provide ‚Äî more data = higher confidence recommendations.

=== 4. Answer Adaptive Questions ===

The plugin asks 3‚Äì8 targeted questions based on what it detects:

* **Strategic:** Where do you want to deploy? What are your availability requirements? Expected traffic patterns?
* **Service-specific:** Do your Cloud SQL databases need read replicas? Are your GKE workloads stateful?
* **AI-specific:** What's your peak token throughput? Do you need streaming responses?

=== 5. Receive Your Migration Plan ===

The plugin generates:

* ‚úÖ **GCP ‚Üí AWS service mapping** with rationale for each recommendation
* ‚úÖ **Production-ready Terraform** ‚Äî run {{code}}terraform init && terraform apply{{/code}}
* ‚úÖ **Architecture diagrams** ‚Äî before (GCP) and after (AWS)
* ‚úÖ **5-phase migration runbook** ‚Äî with rollback procedures at each phase
* ‚úÖ **Cost comparison** ‚Äî monthly/annual GCP vs. AWS with optimization tips
* ‚úÖ **Code change guidance** ‚Äî specific diffs showing SDK/API changes
* ‚úÖ **AWS credit eligibility** ‚Äî Activate, IW Migrate, or MAP qualification

Here's what a typical plugin session looks like for a startup running GKE, Cloud SQL (PostgreSQL), and Cloud Run on GCP:

**Prompt:**

{{code}}

Cloud SQL PostgreSQL databases, and Cloud Run services. Analyze my
Terraform files and create a migration plan.
{{/code}}

**Plugin detects:**

* 2 GKE clusters (production + staging)
* 3 Cloud SQL PostgreSQL instances (primary + 2 read replicas)
* 5 Cloud Run services (HTTP APIs)
* Cloud Storage buckets for static assets
* Pub/Sub topics for async messaging

**Plugin recommends:**

|=GCP|=‚Üí AWS|=Rationale

|Cloud Run|AWS Fargate|2 services (WebSocket, long-running)

== Example: Migrating OpenAI to Amazon Bedrock ==

**Prompt:**

{{code}}
Migrate my OpenAI API usage to Amazon Bedrock. I use GPT-4o for
customer support and GPT-3.5-turbo for classification tasks.
{{/code}}

**Plugin recommends:**

|=Current|=‚Üí Bedrock|=Rationale
|GPT-4o (support)|Claude Sonnet 4|Quality parity, lower cost at volume
|GPT-3.5-turbo (classification)|Claude Haiku|3x cheaper, sufficient for classification

**Code change example generated:**

{{code language="python"}}
# Before (OpenAI)
from openai import OpenAI
client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": prompt}]
)

# After (Bedrock)
import boto3
client = boto3.client("bedrock-runtime")
response = client.converse(
    modelId="anthropic.claude-sonnet-4-20250514-v1:0",
    messages=[{"role": "user", "content": [{"text": prompt}]}]
)
{{/code}}

== Troubleshooting ==

**Q: Plugin not detecting my Terraform files?**
A: Ensure {{code}}.tf{{/code}} files are in your workspace root or subdirectories. The plugin scans the current workspace recursively.

**Q: Can I run the plugin on a monorepo?**
A: Yes. Open the repo root in your IDE and the plugin will scan all subdirectories for infrastructure code.

**Q: Cost estimates seem off?**
A: Provide GCP billing exports for higher accuracy. Without billing data, estimates are based on Terraform resource specifications which may not reflect actual usage.

**Q: Plugin asks too many questions?**
A: Provide more data upfront (Terraform + billing). The more the plugin can infer, the fewer questions it asks.

**Q: How do I update the plugin?**
A: Kiro/Cursor: Extensions panel ‚Üí AWS Startup Advisor ‚Üí Update. Claude Code: Automatic. Codex: {{code}}/plugin update migration-to-aws@agent-plugins-for-aws{{/code}}.
